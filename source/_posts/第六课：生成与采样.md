> 本课是 CS336 Assignment 1 理论准备系列的第六课。前五课里，我们已经把“模型怎么训练出来”这条主线搭完整了。现在我们来学习另一条非常关键的主线：一个训练好的语言模型，到底是怎样生成文本的？

---

## 1. 训练和生成不是一回事

这是理解语言模型最重要的分界线之一。

### 1.1 训练时发生了什么

训练时，模型拿到的是完整的真实 token 序列。例如：

```text
[BOS, I, love, cats, EOS]
```

模型在每个位置都看到“真实前文”，然后预测下一个 token。

例如：

- 看到 `[BOS]`，预测 `I`
- 看到 `[BOS, I]`，预测 `love`
- 看到 `[BOS, I, love]`，预测 `cats`

这叫做：

> teacher forcing

也就是：

> 在训练时，前文不是模型自己生成的，而是数据集给你的真实答案。

### 1.2 生成时发生了什么

生成时就完全不同了。

模型只知道一个起点，比如：

```text
[BOS]
```

然后：

1. 预测下一个 token
2. 把它自己刚刚预测出来的 token 接到输入后面
3. 再预测下一个 token
4. 重复这个过程

也就是说：

> 生成时，模型的未来输入来自它自己的过去输出。

这就是自回归生成（autoregressive generation）。

---

## 2. 为什么训练时看起来很稳，生成时却可能胡说八道

这个问题非常关键。

### 2.1 因为训练时每一步的输入都是真实的

在训练中，就算模型前一步本来会预测错，它下一步仍然能看到真实 token。

所以训练过程更像：

- 每一步都有人把你扶正

### 2.2 但生成时没有人扶正

生成时，如果模型某一步选错了 token，那么后续输入就已经偏了。

而后面的所有预测都会建立在这个错误前文之上。

这会导致：

- 错误逐步积累
- 输出越来越怪
- 甚至语义完全跑偏

这也是为什么语言模型生成看起来比训练更“脆弱”。

---

## 3. 生成流程的最小版本

### 3.1 初始输入

你先给模型一个 prompt：

```text
"The cat"
```

经过 tokenizer 以后变成：

```text
[464, 3797]
```

### 3.2 模型前向

把这串 token 输入模型，得到 logits：

```text
logits.shape = (sequence_length, vocab_size)
```

如果 batch 维度也在，就是：

```text
(batch_size, sequence_length, vocab_size)
```

### 3.3 只取最后一个位置

注意：生成时，我们真正关心的是最后一个位置的输出。

为什么？

因为：

- 前面的 token 都已经确定了
- 我们现在是要预测“下一个 token”

所以会取：

```text
logits[:, -1, :]
```

也就是：

> 当前序列最后一个位置，对整个词表的打分。

### 3.4 logits -> 概率

对最后一个位置的 logits 做 softmax：

```text
probs = softmax(logits_last)
```

于是就得到一个概率分布：

- token A 的概率是多少
- token B 的概率是多少
- token C 的概率是多少

### 3.5 选下一个 token

然后按照某种规则从这个分布里选出一个 token：

- greedy：选概率最大的
- sampling：按概率随机采样
- top-k：只在前 k 个里采样
- top-p：只在累计概率达到 p 的集合里采样

### 3.6 拼回输入，继续生成

把新 token 接回序列：

```text
[464, 3797, new_token]
```

然后继续下一轮。

这就是完整的逐 token 生成过程。

## 4. Greedy Decoding

### 4.1 定义

greedy decoding 的规则最简单：

> 每一步都选概率最大的 token。

也就是：

```text
next_token = argmax(probs)
```

### 4.2 优点

- 简单
- 稳定
- 可重复
- 不引入随机性

### 4.3 缺点

它太保守了。

模型每一步都只走当前最优路径，容易导致：

- 文本很死板
- 重复
- 缺少多样性

举个例子，模型可能一直倾向生成最常见的句式，结果看起来像“平均答案”。

### 4.4 什么时候适合用 greedy

适合：

- 你想调试模型
- 想复现固定输出
- 想快速验证模型大方向有没有问题

不适合：

- 想让输出更自然、更有变化

---

## 5. Sampling

### 5.1 定义

sampling 不是选最大概率，而是按概率随机抽样。

例如：

```text
probs = [0.7, 0.2, 0.1]
```

那就表示：

- 第一个 token 有 70% 概率被选中
- 第二个 token 有 20%
- 第三个 token 有 10%

### 5.2 优点

- 能生成更多样的输出
- 不会总是走单一路径

### 5.3 缺点

- 结果有随机性
- 有时会抽到低质量 token
- 温度太高时可能“发散”

所以 sampling 的关键不是“要不要随机”，而是：

> 怎样控制随机性。

---

## 6. Temperature

### 6.1 它调的是什么

temperature 是在 softmax 前对 logits 做缩放：

```text
scaled_logits = logits / temperature
```

然后再：

```text
probs = softmax(scaled_logits)
```

### 6.2 直觉理解

temperature 决定分布“尖锐”还是“平坦”。

#### 当 temperature < 1

例如 `0.7` 或 `0.5`：

- 大的 logits 会变得相对更大
- 小的 logits 会变得相对更小
- 概率分布更尖锐

结果：

- 模型更保守
- 更像 greedy
- 输出更稳定但多样性下降

#### 当 temperature = 1

- 就是原始分布

#### 当 temperature > 1

例如 `1.2`、`1.5`、`2.0`：

- 概率分布更平坦
- 低概率 token 更容易被选中

结果：

- 文本更有创造性
- 但更容易出奇怪输出

### 6.3 极端情况

#### temperature -> 0

会越来越接近：

```text
argmax
```

也就是 greedy。

#### temperature 非常大

概率会越来越平均，最后接近乱抽。

### 6.4 实际经验

常见取值：

- `0.7`：更稳、更保守
- `1.0`：默认
- `1.2`：更发散、更有变化

## 7. Top-k Sampling

### 7.1 定义

top-k 的规则是：

> 先把概率最高的前 k 个 token 留下，其余全部砍掉；然后只在这 k 个里重新归一化并采样。

### 7.2 例子

假设词表概率如下：

```text
A: 0.40
B: 0.25
C: 0.15
D: 0.10
E: 0.05
F: 0.05
```

如果 `top_k = 3`，就只保留：

```text
A, B, C
```

其余 `D/E/F` 概率清零。

然后对 `A/B/C` 重新归一化：

```text
A: 0.40 / 0.80 = 0.50
B: 0.25 / 0.80 = 0.3125
C: 0.15 / 0.80 = 0.1875
```

### 7.3 优点

- 防止抽到非常离谱的长尾 token
- 比纯 sampling 更稳
- 比 greedy 更有变化

### 7.4 缺点

- `k` 是固定阈值，比较粗糙
- 某些时候前 50 个 token 已经很多，某些时候又太少

---

## 8. Top-p / Nucleus Sampling

### 8.1 定义

top-p 的规则是：

> 按概率从高到低排序，取最小的一个 token 集合，使它们的累计概率达到 p，然后只在这个集合里采样。

### 8.2 为什么比 top-k 更自适应

因为它不是固定取前 `k` 个，而是根据当前分布形状自适应决定保留多少个 token。

例如：

#### 情况 1：分布非常尖锐

```text
A: 0.85
B: 0.10
C: 0.03
...
```

如果 `top_p = 0.9`，可能只保留：

```text
A, B
```

#### 情况 2：分布比较平坦

```text
A: 0.20
B: 0.18
C: 0.15
D: 0.14
E: 0.12
...
```

同样 `top_p = 0.9`，可能要保留很多 token。

### 8.3 优点

- 比 top-k 更灵活
- 往往生成质量更自然

### 8.4 缺点

- 实现稍复杂
- 参数 `p` 也需要调

---

## 9. 为什么生成时只取最后一个位置

这是一个很基础但很容易混淆的点。

假设当前输入长度是 5：

```text
x = [x1, x2, x3, x4, x5]
```

模型输出 logits 形状：

```text
(5, vocab_size)
```

其中每个位置都对应一个“预测下一个 token”的分布：

- 第 1 个位置的输出，是预测 `x2`
- 第 2 个位置的输出，是预测 `x3`
- ...
- 第 5 个位置的输出，是预测“还没出现的下一个 token”

所以在生成时，你真正需要的是：

> 最后一个位置的 logits。

因为那才对应“当前上下文之后”的下一个 token。

---

## 10. 为什么要有停止条件

如果你不给停止条件，模型就会一直生成下去。

常见停止条件有：

### 10.1 生成到 special token

例如：

- `<|endoftext|>`
- `EOS`

这表示：

> 模型认为当前序列应该结束了。

### 10.2 达到最大长度

例如：

```text
max_new_tokens = 100
```

这是一种安全保护，防止无限生成。

### 10.3 任务特定停止符

有些任务还会规定：

- 遇到换行结束
- 遇到某个分隔符结束

## 11. 训练时和生成时的一个本质差别

这个差别非常重要，值得单独强调。

### 11.1 训练时是并行的

训练时你有完整序列，可以一次性把整段喂进去：

```text
[x1, x2, x3, x4, x5]
```

模型在一个前向里就能同时得到：

- 位置 1 的预测
- 位置 2 的预测
- 位置 3 的预测
- ...

所以训练效率高。

### 11.2 生成时是串行的

生成时你没有未来 token，只能：

- 先生成第 1 个新 token
- 再把它接回去
- 再生成第 2 个新 token

所以生成是天然逐步、串行的。

这也是为什么：

> 训练通常比推理吞吐更高，而单条生成常常更慢。

## 12. KV Cache 是为了解决什么问题

虽然 A1 第六课理论里你可以先不深入实现细节，但概念上应该知道它干什么。

### 12.1 没有 KV cache 时

每次生成一个新 token，都把整个前缀重新过一遍 attention。

例如生成第 100 个 token 时，又把前 99 个 token 的 K/V 全重新算一遍。

这很浪费。

### 12.2 有 KV cache 时

前面 token 的 key/value 直接缓存起来。

下一步只需要：

- 计算新 token 的 query / key / value
- 用 query 去和缓存的 key 做 attention
- value 也直接从缓存里取

这样生成会快很多。

### 12.3 为什么训练里一般不显式用它

因为训练时常常整段并行算完，很多框架把它隐含在矩阵运算里。

但在生成里，它是非常关键的优化。

## 13. A1 里生成部分和前面知识怎么连起来

你前面已经学过：

- tokenizer：把 prompt 变成 token ids
- Transformer：把 token ids -> logits
- softmax：把 logits -> 概率

第六课就是把这三件事串起来：

```text
prompt
-> tokenizer.encode
-> model forward
-> 取最后一个位置 logits
-> softmax
-> decoding / sampling
-> append 新 token
-> 循环
-> tokenizer.decode
```

所以生成不是一个新模型，而只是：

> 对同一个训练好的 next-token 模型，反复调用“预测下一个 token”的过程。

---

## 14. 最常见的生成问题

### 14.1 一直重复

例如：

```text
the the the the...
```

常见原因：

- greedy 太保守
- temperature 太低
- 模型本身训练不够好

### 14.2 太发散

例如生成大量乱七八糟的词。

常见原因：

- temperature 太高
- top-k / top-p 约束不够

### 14.3 很快结束

可能是：

- EOS token 概率过高
- prompt 太短
- 模型训练偏向短样本

### 14.4 看起来“语法没问题但内容很假”

这其实是语言模型的典型特征：

- 它学的是统计上的合理续写
- 不是事实保证器

## 15. 本课和 microgpt 的关系

Karpathy 的 microgpt 正好就是第六课的一个非常好的最小例子。

它的生成流程就是：

- 从 BOS 开始
- 前向得到 logits
- softmax 成概率
- 按温度采样一个 token
- 如果是 BOS 就停止
- 否则继续

你可以把 microgpt 看成：

> 第三课（Transformer） + 第四课（训练） + 第六课（生成）的最小整合版。

## 16. 本课总结

请你记住下面这些结论：

- 训练和生成不是一回事：训练用真实前文，生成用模型自己的输出
- 生成时每一步只关心最后一个位置的 logits
- greedy decoding 是每步选最大概率 token
- sampling 是按概率随机抽样
- temperature 控制分布尖锐程度
- top-k 和 top-p 是为了防止抽到太离谱的低概率 token
- 生成必须有停止条件
- 生成本质上是反复调用同一个 next-token 模型
